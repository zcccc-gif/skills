# 测试驱动开发（TDD）

## 理念

**核心原则**：测试应该通过公共接口验证行为，而非实现细节。代码可以完全改变，测试不应该改变。

**好测试**是集成风格的：通过公共 API 真实运行代码路径。描述**系统做什么**，而非**如何做**。一个好的测试像规格说明——"用户可以用有效购物车结账"告诉你存在什么能力。这些测试在重构后仍然有效，因为它们不关心内部结构。

**坏测试**耦合到实现。它们模拟内部协作者、测试私有方法、或通过外部方式验证（如直接查询数据库而非使用接口）。警告信号：重构后测试失败，但行为没变。如果重命名内部函数后测试失败，那些测试测试的是实现，不是行为。

参见 [tests.md](tests.md) 和 [mocking.md](mocking.md)。

## 反模式：水平切片

**不要先写所有测试，再写所有实现。** 这是"水平切片"——把 RED 当成"写所有测试"，GREEN 当成"写所有代码"。

这会产生**垃圾测试**：

- 批量写的测试测试的是**想象的行为**，不是**实际的行为**
- 最终测试的是**东西的形状**（数据结构、函数签名）而非用户面向的行为
- 测试变得对真实变化不敏感——行为坏了测试通过，行为没问题测试失败
- 超出预期，在理解实现前就承诺了测试结构

**正确方法**：垂直切片，追踪子弹。一个测试 → 一个实现 → 重复。每个测试响应从上一个循环中学到的东西。因为刚写了代码，完全知道什么行为重要、如何验证。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流程

### 1. 计划

探索代码库时，使用项目的领域术语使测试名称和接口词汇匹配，并尊重你正在触碰区域的 ADR。

写任何代码前：

- [ ] 确认用户需要什么接口更改
- [ ] 确认用户要测试哪些行为（按优先级）
- [ ] 识别[深度模块](deep-modules.md)的机会
- [ ] 设计[可测试性](interface-design.md)的接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的批准

问："公共接口应该是什么样？哪些行为最重要需要测试？"

**不能测试一切。** 确认用户哪些行为最重要。在关键路径和复杂逻辑上集中测试，而非每个可能的边缘情况。

### 2. 追踪子弹

写一个测试确认系统的一件事：

```
RED:   为第一个行为写测试 → 测试失败
GREEN: 写最少的代码通过 → 测试通过
```

这是你的追踪子弹——证明路径端到端有效。

### 3. 增量循环

每个剩余行为：

```
RED:   写下个测试 → 失败
GREEN: 最少代码通过 → 通过
```

规则：

- 一次一个测试
- 只够通过当前测试的代码
- 不预测未来测试
- 保持测试聚焦在可观察行为

### 4. 重构

所有测试通过后，寻找[重构候选](refactoring.md)：

- [ ] 提取重复
- [ ] 加深模块（将复杂 behind 简单接口）
- [ ] 在自然处应用 SOLID 原则
- [ ] 考虑新代码揭示了现有代码的什么
- [ ] 每步重构后运行测试

**RED 时绝不重构。** 先到 GREEN。

## 每循环检查清单

```
[ ] 测试描述行为，而非实现
[ ] 测试只用公共接口
[ ] 测试能在内部重构后存活
[ ] 代码对这个测试是最少的
[ ] 没有添加投机功能
```
