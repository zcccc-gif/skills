# 诊断（Diagnose）

困难 bug 的纪律。除非有明确理由，否则跳过各阶段。

探索代码库时，使用项目的领域术语获取相关模块的清晰心智模型，并检查你正在触碰区域的 ADR。

## 阶段 1 — 构建反馈循环

**这就是技能。** 其他一切都是机械的。如果你有一个快速、确定性、可 agent 运行的 pass/fail 信号来检测 bug，你会找到原因——二分、假设测试和插桩都只是消耗那个信号。如果你没有，再怎么看代码也救不了你。

在这里投入不成比例的努力。**要激进。要有创意。不要放弃。**

### 构建反馈循环的方法 — 大致按此顺序尝试

1. **失败测试** 在任何触及 bug 的接缝处 — 单元、集成、e2e。
2. **Curl / HTTP 脚本** 针对运行中的开发服务器。
3. **CLI 调用** 带有 fixture 输入，对 stdout 与已知良好快照 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）— 驱动 UI，断言 DOM/console/network。
5. **重放捕获的跟踪。** 将真实网络请求/有效载荷/事件日志保存到磁盘；通过代码路径隔离重放。
6. **一次性工具。** 启动触及 bug 代码路径的最小系统子集（一个服务，mocked deps）用单个函数调用练习。
7. **属性/模糊循环。** 如果 bug 是"有时错误输出"，运行 1000 个随机输入并寻找失败模式。
8. **二分工具。** 如果 bug 出现在两个已知状态之间（commit、数据集、版本），自动化"在状态 X 启动，检查，重复"以便 `git bisect run` 它。
9. **差分循环。** 用旧版本 vs 新版本（或两个配置）运行相同输入并 diff 输出。
10. **HITL bash 脚本。** 最后手段。如果必须由人点击，用 `scripts/hitl-loop.template.sh` 驱动他们使循环仍然是结构化的。捕获的输出反馈给你。

构建正确的反馈循环，bug 就 90% 修复了。

### 迭代循环本身

把循环当作产品。一旦有了_一个_循环，问：

- 能让它更快吗？（缓存设置，跳过无关初始化，缩小测试范围。）
- 能让信号更尖锐吗？（断言特定症状，不是"没崩溃"。）
- 能让它更确定性吗？（固定时间，seed RNG，隔离文件系统，冻结网络。）

30 秒的不稳定循环比没有循环好不了多少。2 秒的确定性循环是调试超能力。

### 非确定性 bug

目标不是干净的复现而是**更高的复现率**。循环触发 100×，并行化，增加压力，缩小时间窗口，注入 sleep。50% flake 的 bug 是可调试的；1% 不是 — 不断提高速率直到可调试。

### 当真的无法构建循环时

明确说出来。列出你尝试过的。询问用户：（a）访问任何能复现的环境，（b）捕获的工件（HAR 文件、日志转储、core dump、带时间戳的屏幕录制），或（c）添加临时生产插桩的许可。在有循环之前**不要进入假设阶段**。

在有循环之前不要进入阶段 2。

## 阶段 2 — 复现

运行循环。看着 bug 出现。

确认：

- [ ] 循环产生用户描述的失败模式 — 不是恰好在附近的不同的失败。错 bug = 错修复。
- [ ] 失败在多次运行中可复现（或者，对于非确定性 bug，以足够高的速率可复现以进行调试）。
- [ ] 你捕获了确切的症状（错误消息、错误输出、慢时序）以便后续阶段验证修复实际解决了它。

在复现 bug 之前不要继续。

## 阶段 3 — 假设

在测试任何之前生成 **3-5 个排名假设**。单一假设生成锚定在第一个 plausible 想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 将使 bug 消失 / <改变 Z> 将使它更糟。"

如果不能陈述预测，假设就是一个 vibe — 丢弃或使其更锐利。

**在测试之前向用户展示排名列表。** 他们通常有领域知识立即重新排名（"我们刚部署了对 #3 的更改"），或者知道他们已经排除的假设。便宜 checkpoint，大时间节省。不要阻塞它 — 如果用户 AFK 就按你的排名继续。

## 阶段 4 — 插桩

每个探针必须映射到阶段 3 的特定预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查** 如果环境支持。一个断点胜过十个日志。
2. **有针对性的日志** 在区分假设的边界上。
3. 永远不要"记录一切然后 grep"。

**给每个调试日志加标签**，例如 `[DEBUG-a4f2]`。最终清理变成单个 grep。未标签日志保留；标签日志死亡。

**性能分支。** 对于性能回归，日志通常是的。建立基线测量（时序工具、`performance.now()`、profiler、查询计划），然后二分。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在修复**之前**编写回归测试 — 但只有存在**正确的接缝**时。

正确的接缝是测试在调用站点真实发生 bug 模式的测试。如果唯一可用的接缝太浅（单调用者测试而 bug 需要多个调用者，无法复制触发 bug 的链的单元测试），那里的回归测试给出虚假信心。

**如果没有正确的接缝存在，那本身就是发现。** 注意它。代码库架构阻止 bug 被锁定。将其 flag 给下一阶段。

如果存在正确的接缝：

1. 将最小化复现变成在该接缝处的失败测试。
2. 看着它失败。
3. 应用修复。
4. 看着它通过。
5. 针对原始（未最小化）场景重新运行阶段 1 循环。

## 阶段 6 — 清理 + 事后分析

在宣布完成前必须：

- [ ] 原始复现不再复现（重新运行阶段 1 循环）
- [ ] 回归测试通过（或记录缺少接缝）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（`grep` 前缀）
- [ ] 一次性原型已删除（或移动到明确标记的调试位置）
- [ ] 正确假设在 commit / PR 消息中说明 — 以便下一个调试者学习

**然后问：什么本可以防止这个 bug？** 如果答案涉及架构更改（没有好的测试接缝、 tangled 调用者、隐藏耦合）将 specifics 交给 `/improve-codebase-architecture` 技能。在修复之后提出推荐，**不是之前** — 你现在有比开始时更多的信息。
